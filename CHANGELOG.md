# История версий

## 0.29.0 — 2026-07-10

Модель данных под реальный процесс: отходы вводятся справками-актами,
из них считаются все отходные формы; по воздуху — источник НДВ/ООС.

- **Справки-акты на отходы — первичный ввод** (`WasteAct`): наименование,
  ФККО, класс, масса, объём/плотность, **вид обращения** (утилизация/
  обезвреживание/размещение/хранение), **перевозчик**, **приёмщик/полигон**,
  дата. Соответствует реальным справкам (сверено с `Справки-2025.xls`).
- **Движение считается из актов** (`ecodoc/core/waste_agg.py`): агрегация по
  ФККО → образовано/передано (в т.ч. на захоронение/хранение) + список
  получателей для Приложения 3 журнала №1028 / кадастра / формы III кат.
  Акты первичны: заданы акты → движение пересчитывается из них; актов нет —
  остаётся заполненное вручную. Все отходные формы (1028, 2-ТП отходы,
  кадастр, раздел отходов декларации) берут данные из одного места.
- **По воздуху** у вещества добавлен признак источника (`Pollutant.source` —
  «НДВ»/«ООС»); в форме данных подсказка «массы берутся из проекта НДВ или
  раздела ООС».
- GUI «Данные»: таблица **«Справки-акты на отходы»** (первичный ввод) + блок
  движения (заполняется из актов) + подсказка по воздуху. `serialize`
  разбирает `waste_acts` и авто-считает движение при загрузке.

## 0.27.0 — 2026-07-10

Контур подачи в ЛКПП РПН + сверка XML 2-ТП (отходы) с тем, что реально
принимает кабинет (по итогам разведки: формат «Копии» = DATA_PACKET_NI).

- **Сверка XML под реального плательщика (в т.ч. ИП):**
  - `FINDIVID=true` для ИП (по 12-значному ИНН), КПП пустой, должность
    подписанта — «Индивидуальный предприниматель» (было «Генеральный
    директор»); `Organization.is_individual` / `official_title`.
  - `NOT_REG_OBJ` больше не захардкожен `true`: при указанном коде объекта
    НВОС (напр. `41-0247-000123-П`) выводится `REG_NUMBER` и `NOT_REG_OBJ=false`
    — устраняет отказ «объект не стоит на учёте».
  - `TP2_TR_DISP` = передано **на захоронение** (было ошибочно «всего
    передано» = `transferred`); в печати графа «для захоронения» — тоже.
  - **Preflight-валидация** перед подачей: обязательные ОГРН/ОГРНИП (с
    контрольной суммой), формат ОКТМО (8/11 цифр, не ОКАТО), наличие кода
    объекта НВОС, проверка ИНН.
- **Команда `ecodoc submit <форма>`** и модуль `ecodoc/submit`: собирает
  папку пакета `out/<ИНН>_<форма>_<год>/` — XML (DATA_PACKET_NI) + печать +
  файлы МЧД + **`ЧЕКЛИСТ.md`** (результат preflight, пошаговая инструкция
  загрузки в ЛКПП «Новый отчёт → Импорт XML → подписать УКЭП», памятка по
  МЧД с проверкой кода полномочия `RPNDZ_REPORT`).
- **Автоотправку по API РПН НЕ реализуем** намеренно: требует индивидуальной
  заявки на OD@rpn.gov.ru под каждый ОГРН, закрытой спецификации endpoints и
  подписи УКЭП+МЧД через КриптоПро — вне оффлайн-приложения. ЭКО.DOC —
  «генератор пакета», подача и подпись остаются за пользователем.

- GUI: клик по организации выбирает её (и единственную площадку сразу);
  у организаций без площадок — подпись «нет площадок».
- **Загрузка папки целиком** («Папку целиком…»): берутся все поддерживаемые
  файлы (pdf/doc/docx/xls/xlsx/jpg/png/tif/txt/xml/rtf), лишнее
  пропускается с пометкой.
- **Распределение файлов при приёме**: для каждого файла — что это за
  документ (акт утилизации, протокол КХА, паспорт отхода, выписка ЕГРЮЛ,
  лицензия, журнал учёта и т.д.), какие данные из него можно взять и
  каким формам он пригодится; в конце — чего не хватает.
